# ドローン産業トレンド分析 — 無人航空機飛行計画データ（2025年度）

国土交通省 Project LINKS が公開する『無人航空機飛行計画データ（2025年度）』を集計し、**ドローンの飛行目的・機体構成・操縦資格／機体認証の保有状況**と、その**月次推移**を可視化する静的ダッシュボードです。研究・業界分析・社内ナレッジ用途を想定しています。

**公開サイト:** https://shinyanakashima.github.io/MLIT-LINKS-uav-pulse/ （日本語／English 切替対応）

> *A static dashboard visualizing the purpose, aircraft mix, and pilot/airworthiness credentials of Japanese UAV flight plans (MLIT Project LINKS, FY2025), with monthly trends. Bilingual (JA/EN).*

> ⚠️ **データの性質（必読）**
> 本データは飛行**計画（申請）**であって、実際の飛行や運航実態ではありません。元データは紙資料のスキャンから機械抽出されたもので、**完全性・正確性は公式に保証されていません**。あくまで傾向把握の参考としてご利用ください。

## 主な集計結果（2024年7月〜2025年6月）

| 指標 | 値 |
|---|---|
| 飛行計画 総数（重複排除） | 約 **65.5 万件** |
| 登録機体数（累計） | 約 **89.4 万機** |
| 元データ行数 | 約 **297 万行** |
| 包括申請（多目的を一括申請）の割合 | 約 **9.9%** |

- 申請件数は対象期間で**右肩上がり**に増加（制度の普及を示唆）。
- 飛行目的は **空撮** と **農林水産業** が二大用途。
- 機体は **回転翼（マルチローター）** が大多数。
- 操縦資格は **技能証明（二等）** が中心で、機体認証はごく少数。

数値はダッシュボード上のグラフおよび `docs/data/meta.json` で確認できます。

## ダッシュボードの機能

- 月次の飛行計画件数推移
- 飛行目的の構成（業務）と、主要目的の月次推移
- 機体の種類・製造区分・改造の有無の構成
- 操縦資格／機体認証の保有率と、その月次推移
- 飛行空域（DID・150m以上など）と飛行方法（夜間・目視外など）の構成
- 出発地（都道府県）別の件数上位
- **日本語／英語の言語切替**（グラフのカテゴリ名も翻訳）

## 分析の単位（重要）

元データは1行＝(飛行計画 × 機体 × 飛行エリア) の組み合わせで、**1つの飛行計画が多数の行に展開**されます（1計画で数千行に及ぶ例もあり）。そのまま行を数えると一部の大型計画に分布が偏るため、本集計では単位を分けています。

- **飛行計画単位**（重複排除）… 飛行目的・操縦資格／機体認証・飛行空域・飛行方法・出発地
- **機体単位**（重複排除）… 機体の種類・製造区分・改造の有無

## リポジトリ構成

```
etl/
  manifest.csv               対象16ファイル（リソースIDと対象月）
  download.py                CKAN 経由で GeoJSON を data/raw/ に取得
  aggregate.py               ストリーム解析で集約 JSON を生成
docs/                        GitHub Pages で配信する静的サイト（公開物）
  index.html / style.css / app.js
  vendor/chart.umd.min.js    Chart.js（同梱。実行時の外部CDN不要）
  data/
    summary.json             全体＋月次の集計値（サイトが読込む。約24KB）
    meta.json                総件数・生成日・出典表記
.github/workflows/pages.yml  main への push で docs/ を自動デプロイ
data/raw/                    取得した生 GeoJSON（.gitignore 済み・合計約9GB）
```

リポジトリに含まれるのは集約済み JSON（数十KB）のみで、生データ（約9GB）は含みません。

## データの再生成手順

**動作要件:** Python 3.9+、`ijson`、空きディスク約 10GB、ネットワーク（geospatial.jp への到達）。

```bash
# 1. 生データ（全16ファイル・合計約9GB）を取得
python3 etl/download.py

# 2. 依存をインストールして集約 JSON を生成（geometry は読み飛ばす）
pip install ijson
python3 etl/aggregate.py
```

`docs/data/summary.json` と `docs/data/meta.json` が更新されます（処理時間の目安は4〜5分）。新しい対象月が公開された場合は `etl/manifest.csv` に行を追加して再実行してください。

## デプロイ

`main` ブランチへの push をトリガーに、GitHub Actions（`.github/workflows/pages.yml`）が `docs/` を GitHub Pages へ自動デプロイします（`actions/configure-pages` が Pages を自動有効化）。集計済み JSON と同梱 Chart.js のみで動く完全な静的配信のため、サーバーサイド処理や API キーは不要です。

GitHub Pages を使えない場合（private リポジトリ等）は、`docs/` 配下一式を任意の静的ホスティング（例: ConoHa WING）にそのまま配置すれば動作します。

## 集計の前処理方針

公開データに見られる癖を、次のように補正しています。

| 課題 | 対応 |
|---|---|
| フィールド名の表記ゆれ（末尾スペース、`機体認証(⼀種)` の部首字 U+2F00、全角半角混在） | キー名を NFKC 正規化 + trim してから参照（生キーをハードコードしない） |
| 日時データの品質が低い（年跨ぎ等） | 月次は「ファイル＝対象月」を信頼単位とし、`飛行日時_*` は時系列に使わない |
| 包括申請ノイズ（飛行目的を多数同時申請） | 業務目的フラグが閾値（既定6）以上の計画を「包括申請」とし、目的別構成から除外して別掲 |
| 1計画が多数行に展開され分布が偏る | 飛行計画単位／機体単位で重複排除（上記「分析の単位」参照） |
| 規模が重い（geometry 含め約9GB） | `ijson` で `features.item.properties` のみをストリーム解析し、Polygon は読み込まない |
| 数値諸元が文字列・欠損あり | null 安全に解析 |
| 出発地は市区町村重心レベルに秘匿化済み | 都道府県粒度までの集計にとどめ、粒度を上げる二次加工はしない |
